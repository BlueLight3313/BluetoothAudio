Re: Basic bluetooth app in C/C++

Tue Oct 19, 2021 4:12 pm

HCI commands are at the lowest level of communication with the bluetooth adapter. The bluez library funtions in libbluetooth-dev such as connect() and accept() send and receive these HCI packets to perform their tasks. If you look in the reference section of the btferret documentation, you will see the full horror of the HCI sequences that are required. Somewhere along the line I think the library functions available to C have been neglected and they do not work reliably. The mit.edu code examples will usually connect to another Pi running mirror code, but often fail when connecting to Windows or Android devices. It is never clear why this happens, and impossible to fix because the guts of the library functions are inaccessible. Often, bluez will send a disconnect instruction for no apparent reason - especially with LE devices. Eventually, I ran out of patience with libbluetooth-dev, extracted the minimum workable HCI sequences and put them in the btferret library btlib. So btferret/btlib is a C bluetooth interface that is a replacement for bluez/libbluetooth-dev, with its own connect and accept methods which work (for me anyway) when bluez will not.

My 2 cents:
D-bus is the recommended way to communicate with BlueZ. I would think there's lots of C code out there for talking to D-bus. Then you'll just need to know what BlueZ exposes, and the BlueZ interfaces are well documented. Falling back to HCI seems to be taking a big step backwards; BlueZ is here for a reason. By the way, there's just a small bit of BlueZ code in the kernel, to handle Bluetooth HID devices, mostly.

AnneRanch asked if BLE is really that much different than classic Bluetooth. Yes. It really is that much different.

The first step of most Bluetooth code is to get the local adapter, e.g.to begin sending HCI commands to it. That look's like what hci_get_route is for, in which case you'll need it. It has nothing to do with a Bluetooth manager or any other running application. Bluetooth, if the the daemon is running, is always "on".
